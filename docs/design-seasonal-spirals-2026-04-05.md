## Problem Statement

Scott published seasonal-spirals v0.3.0 to PyPI on April 4, 2026. It's his own open
source Python library that turns any daily time series into a seasonal spiral chart
(interactive Plotly or static matplotlib). He wants to showcase it on stomlins.com.

The prior design doc established a minimal Astro static site with a Projects section
listing external links. seasonal-spirals is a different category of project — a
published library, not a hosted app or article — and the spiral visualization is
genuinely beautiful. It deserves more than a text link.

Min scope: link to repo and PyPI.
Max scope: show multiple plots.
Chosen: a hybrid — static PNG thumbnail on the homepage, dedicated /seasonal-spirals
page with multiple interactive charts.

---

## What Makes This Cool

seasonal-spirals is Scott's own library on PyPI. Most "data projects" in a portfolio
are one-off dashboards. A published, installable Python package is a different signal.
The spiral chart is immediately striking and does its own selling.

The approach: let the image earn the click. Static PNG on the homepage — fast, no JS.
The /seasonal-spirals page delivers the full interactive experience for people who want it.

---

## Constraints

- Follows the approved design system from prior doc (dark, Plus Jakarta Sans, 680px max-width)
- Homepage must stay under 500ms cold load — no Plotly.js on the main page
- Charts and images generated locally from Python and committed to repo — no build-time deps
- Static Astro pages only — no server-side rendering or Python runtime on Pi

---

## Premises

1. A static PNG thumbnail on the homepage is better than an embedded interactive chart.
   Homepage stays fast; Plotly.js weight (~3MB) stays off the critical path. **Agreed.**
2. seasonal-spirals warrants a dedicated subpage. It's a published library — it gets
   its own URL that can be linked from PyPI, GitHub, etc. **Agreed.**
3. Pre-exported Plotly HTML artifacts (generated once locally, committed to repo) are the
   right format for the /seasonal-spirals page. Visitors arriving there explicitly want
   charts. **Agreed.**
4. The existing design system applies to the new page — same tokens, same layout. **Agreed.**

## Approaches Considered

### Approach A: PNG only (minimal)
Add seasonal-spirals to projects.yaml with an `image` field. Projects.astro renders
the image if present. Title links to PyPI. No new Astro page.
Effort: XS | Risk: Low | Completeness: 6/10
Pros: Zero new pages. Maximum consistency with existing Projects list.
Cons: Doesn't showcase the library beyond a pretty picture. No URL to share.
     Misses the "multiple plots" goal.

### Approach B: Homepage card + /seasonal-spirals page (CHOSEN)
Static PNG thumbnail in the Projects section card. Title/image links to
`/seasonal-spirals`. The dedicated page has 2-3 pre-exported interactive Plotly charts,
install instructions (pip install seasonal-spirals), and links to PyPI and GitHub.
Effort: S-M (human: 2-3hr / CC: 30min) | Risk: Low | Completeness: 9/10
Pros: Homepage stays fast. Library gets a real URL. Visitors who want the charts
     get them. PyPI page can link back to stomlins.com/seasonal-spirals.
Cons: One new Astro page to maintain. Charts need regenerating when library updates.

### Approach C: Embedded interactive on homepage
Export Plotly figure as standalone HTML, embed inline or as iframe in Projects section.
Effort: M | Risk: Medium | Completeness: 7/10
Pros: Interactive on first load. No click required.
Cons: Plotly.js on homepage (~3MB). Breaks 500ms constraint. Feels inconsistent with
     minimal design system.

## Recommended Approach

**Approach B** — homepage PNG card + `/seasonal-spirals` dedicated page.

### Projects.astro: conditional image render

The title link uses `target="_blank"` for external URLs. For internal URLs (starting
with `/`), omit `target="_blank"` so the user navigates within the same tab.

```astro
---
const isExternal = project.url.startsWith('http');
---

{/* Title link */}
<a
  href={project.url}
  {...isExternal ? { target: '_blank', rel: 'noopener noreferrer' } : {}}
>
  {project.name} ↗
</a>

{/* Optional image */}
{project.image && (
  <a
    href={project.url}
    {...isExternal ? { target: '_blank', rel: 'noopener noreferrer' } : {}}
  >
    <img
      src={project.image}
      alt={`${project.name} spiral chart demo`}
      class="project-img"
      loading="lazy"
      width="640"
      height="400"
    />
  </a>
)}

{/* Optional PyPI / GitHub links */}
{(project.pypi || project.github) && (
  <p class="project-links">
    {project.pypi && <a href={project.pypi} target="_blank" rel="noopener noreferrer">PyPI</a>}
    {project.pypi && project.github && ' · '}
    {project.github && <a href={project.github} target="_blank" rel="noopener noreferrer">GitHub</a>}
  </p>
)}

{/* Tags (no year mixed in — year not displayed in tag row) */}
<p class="project-tags">{project.tags.join(' · ')}</p>
```

### Generating the assets (Python, run locally)

```python
# scripts/generate_spirals.py
# Run with: uv run python scripts/generate_spirals.py
# Requires: seasonal-spirals, plotly, matplotlib, pandas installed
#
# Data: load your time series as a pandas Series with a DatetimeIndex.
# seasonal_spirals does NOT include a data fetcher — bring your own data.
# Example: Wikipedia pageviews via the `mwapi` or `requests` library,
# or any CSV with a date column.

import pandas as pd
from seasonal_spirals import plot_spiral, plot_spiral_static
import plotly.io as pio
from pathlib import Path

# --- Load your data here ---
# df = pd.read_csv('data/six_nations_pageviews.csv', parse_dates=['date'], index_col='date')
# series = df['pageviews']

# Generate PNG thumbnail for homepage (matplotlib)
fig_static = plot_spiral_static(series, title="seasonal-spirals")
Path('public/images').mkdir(parents=True, exist_ok=True)
fig_static.savefig('public/images/seasonal-spirals-demo.png', dpi=150, bbox_inches='tight')
fig_static.close()

# Generate interactive charts for /seasonal-spirals page (Plotly)
Path('public/spirals').mkdir(parents=True, exist_ok=True)
fig1 = plot_spiral(series, title="Chart title")
pio.write_html(
    fig1,
    file='public/spirals/chart-1.html',
    include_plotlyjs='cdn',
    full_html=False   # partial HTML: just the div + script block, not a full page
)
# Verify output size — cdn option should be <100KB per file
# Run: ls -lh public/spirals/ to confirm before committing
```

The exported partial HTML (`full_html=False`) is served as a standalone file loaded via
iframe. **Use iframes, not inline embedding.** See rationale below.

Committed files:
- `public/images/seasonal-spirals-demo.png` — static thumbnail
- `public/spirals/chart-1.html` — Plotly partial HTML (cdn, <100KB expected)
- `public/spirals/chart-2.html` — second chart

The Python script lives in `scripts/generate_spirals.py`. Run it when the library
updates or you want a new chart. Not part of `npm run build`.

### Astro: embedding the chart HTML

**Use iframes.** Simpler, no build-time file reads, no path assumptions, works in all
Astro build environments including CI. The CDN Plotly.js request happens either way.

```astro
<iframe
  src="/spirals/chart-1.html"
  title="Seasonal spiral — Six Nations pageviews"
  width="100%"
  height="500"
  style="border: 1px solid var(--border); border-radius: 6px;"
  loading="lazy"
/>
```

Set `height` based on the chart's aspect ratio after generating it locally — 500px is
a starting point. Add a `<p class="chart-caption">` below each iframe for context.

Do NOT use inline `readFileSync` embedding — the relative path `'public/spirals/chart-1.html'`
is fragile if the build runs from a different working directory (e.g., CI). The iframe
approach has no path assumptions.
